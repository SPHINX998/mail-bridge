# 使用文档

这份文档描述的是系统跑起来之后，你该怎么用它。

## 系统在做什么

当前链路是：

```text
Gmail 新邮件
  -> Pub/Sub
  -> mail-bridge
  -> OpenClaw 判断重要性
  -> 只有重要邮件才发 QQ
```

也就是说：

- 你不需要定时手动扫邮箱
- 系统不是把所有邮件都发给你
- 它只把 OpenClaw 判断为重要的邮件发到 QQ

## 你会收到什么

当前推荐的 QQ 消息一般包含：

- 标题
- 一段摘要
- 一小段正文片段

示例：

```text
🚨 紧急邮件：王总 / 今天18:00前确认付款+合同盖章
摘要：王总发来付款审批和合同终稿，需要你今天18:00前确认是否同意付款并确认合同是否可以盖章。
片段：请你在今天18:00前确认是否同意付款，并确认合同终稿是否可以今天盖章。
```

## 如何判断系统有没有正常工作

### 1. 看健康接口

```powershell
Invoke-RestMethod -Uri 'http://127.0.0.1:8787/healthz'
Invoke-RestMethod -Uri 'http://127.0.0.1:8787/watch/status'
```

### 2. 看是否收到 QQ 提醒

如果发了一个明显紧急的测试邮件，而 QQ 收到了提醒，说明主链路已经打通。

### 3. 看日志

关键日志通常包括：

- `message_classified`
- `notification_sent`

## 如何验证一封邮件是否被系统处理

你可以：

- 查看 `mail-bridge` 日志
- 查看 SQLite 状态库
- 查看 OpenClaw 会话 transcript（如果你的部署启用了持久化）

## 如何纠正“以后怎么提醒”

这是这套系统最重要的部分之一。

你不一定要改代码，也不一定要手写规则。

### 方式 A：在 OpenClaw 对话里直接说

你可以自然地说：

- `以后这类招聘平台营销邮件不要提醒我。`
- `以后王总发来的邮件必须提醒我。`
- `以后合同审批类邮件只发摘要，不要发太长。`
- `以后主题带“付款审批”的邮件优先提醒。`

当前推荐的做法是：

- OpenClaw 通过 `mail-bridge-preferences` skill
- 把这些偏好写回 `mail-bridge`
- 后续同类邮件就会更贴近你的偏好

### 方式 B：直接调用本地接口

查看当前偏好：

```powershell
Invoke-RestMethod -Uri 'http://127.0.0.1:8787/preferences'
```

追加一条备注：

```powershell
Invoke-RestMethod -Method Post -Uri 'http://127.0.0.1:8787/preferences/notes' -ContentType 'application/json' -Body '{"note":"老板发来的付款审批和合同邮件必须提醒"}'
```

### 方式 C：直接使用 CLI

```powershell
python -m mail_bridge.add_preference_note --note "老板发来的付款审批和合同邮件必须提醒"
```

## 推荐的偏好纠正规则

为了让系统学得更稳，建议纠正时尽量说清楚这三件事：

- **对象是谁**：发件人 / 域名 / 关键词 / 主题类型
- **希望怎么处理**：必须提醒 / 不提醒 / 只发摘要 / 只发短提醒
- **原因是什么**：老板 / 截止时间 / 营销邮件 / 太吵

例如：

- `以后包含“发票”的邮件默认提醒，因为涉及报销。`
- `以后某某招聘平台的营销邮件默认不提醒，因为都是推广。`
- `以后合同终稿类邮件默认发摘要，不要发太长正文。`

## 推荐的日常使用方式

### 1. 平时什么都不用做

系统会被动接收 Gmail 事件。

### 2. 只在你觉得提醒不对时纠正

例如：

- 误报：不该提醒却提醒了
- 漏报：该提醒却没提醒
- 提醒内容太长 / 太短

### 3. 持续几天后再看效果

这套系统不是一次性写死规则，而是逐步贴近你的习惯。

## 已知边界

当前第一版默认：

- 只处理 Gmail 主入口
- 只处理新邮件
- 只处理 `INBOX`
- 只做提醒，不做自动回复
- 不自动操作邮件（不自动归档、不自动打标）
- 不把全文直接转发到 QQ

## 适合的使用方式

适合：

- 工作邮件中挑重点
- 审批 / 合同 / 截止时间提醒
- 低打扰邮件筛选

不适合：

- 把它当完整邮件客户端
- 让它代替人工处理所有邮件
- 期待第一次部署就完全零误报零漏报
